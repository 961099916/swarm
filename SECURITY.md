# 安全策略

## 报告安全漏洞

Swarm 项目团队非常重视安全性。如果你发现了安全漏洞，请通过以下方式私密报告，**请勿在公开 Issue 中提交安全漏洞**。

### 报告方式

- 发送邮件至项目维护者（请在 GitHub Security Advisories 页面查看联系方式）
- 使用 GitHub 的 **Private Vulnerability Reporting** 功能（推荐）：
  1. 访问仓库的 Security 标签页
  2. 点击 "Report a vulnerability"
  3. 填写详细信息并提交

### 报告内容

请提供以下信息以帮助我们快速定位和修复：

- 漏洞类型（如 XSS、SQL 注入、权限绕过等）
- 受影响的版本/模块
- 复现步骤（包含代码片段、请求示例等）
- 潜在影响范围
- 建议的修复方案（可选）

### 响应时间

- **24 小时内** — 确认收到报告并启动调查
- **7 天内** — 给出初步评估和修复计划
- **30 天内** — 发布修复版本（复杂漏洞可能延长时间）

## 安全最佳实践

### 生产部署

1. **密钥管理**
   - 使用 `init_secrets.sh` 脚本自动生成高强度密钥（32 字节随机十六进制）
   - 禁止将 `.dev.vars` 或任何包含密钥的文件提交到版本控制
   - 定期轮换 JWT_SECRET 和 INTERNAL_SECRET

2. **物理安全隔离**
   - Gateway 与内部 Worker 之间使用 Service Binding + X-Internal-Key 双重校验
   - 内部 Worker 不直接对外暴露，仅接受来自 Gateway 的请求
   - 禁止在网关层编写业务逻辑，只做路由编排和鉴权

3. **数据脱敏**
   - TraceLogger 自动过滤密码、Token、密钥等敏感字段
   - 禁止在日志中明文打印用户敏感信息（如 wx_open_id）

4. **缓存安全**
   - 管理端操作（角色修改、封禁、强制下线）自动废弃网关缓存
   - 缓存防穿透（空值占位）与防雪崩（TTL 随机偏移）

5. **依赖管理**
   - 定期审查依赖版本，及时更新安全补丁
   - 使用 npm audit 检查已知漏洞

## 已知安全措施

| 措施             | 说明                                                         |
| ---------------- | ------------------------------------------------------------ |
| JWT Token 版本   | 支持 Token 版本校验，管理员可强制用户下线                    |
| RBAC 权限        | 基于角色的访问控制（FREE_USER / VIP_USER / ADMIN）           |
| 封禁机制         | 支持用户封禁与原因记录                                       |
| 审计日志         | 所有管理操作记录审计日志                                     |
| 沙箱执行         | 动态工具执行在安全沙箱中运行                                 |
| CORS 限制        | 仅允许指定的域名访问 API                                     |
| 缓存一致性       | 管理操作实时废弃网关缓存，防止安全延迟漏洞                   |
| 积分限制         | 积分操作有严格的事务边界保证                                 |
