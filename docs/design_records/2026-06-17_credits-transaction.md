# 积分原子性与事务选型

- **状态**: ✅ 已实施
- **日期**: 2026-06-17
- **决策者**: @jiuxia

## 背景

积分（credits）系统是 Swarm 的核心资产，涉及扣减/增加、流水记录、邀请奖励、广告奖励等操作。操作的原子性直接影响用户资产安全。

## 问题

初始实现中使用 `drizzleDb.batch()` 执行积分变更，但 batch 不保证原子性——部分操作成功、部分失败会导致积分不一致。

## 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A: db.transaction()** | ✅ 真正 ACID 原子性；支持 `returning()` 读取数据库端余额 | 需使用 DrizzleD1Database 类型；不支持跨 D1 实例事务 |
| **B: db.batch() (D1 原生)** | ✅ D1 的 batch 是原子提交 | 不适用于 Drizzle ORM 的 batch（非原生 D1 batch） |
| **C: 乐观锁 + 重试** | 高并发下性能好 | 实现复杂度高，需要版本号字段 |

## 决策

选择**方案 A**：积分敏感操作使用 `db.transaction()` + `returning()`。

对于 engine worker 中使用 D1 原生 `db.batch()` 的操作，由于 D1 的 batch API 本身就是原子提交，予以保留。

## 后果

- ✅ 积分操作的原子性得到保障
- ✅ 通过 `returning()` 获取数据库端余额，消除客户端计算竞态
- ⚠️ 需要确保所有积分操作路径都使用事务（已审计）
